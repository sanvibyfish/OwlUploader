## 阿里云 OSS 支持 — 多云抽象实施

### 概述

在现有 Cloudflare R2 基础上新增阿里云 OSS 支持，采用"多云统一抽象"方案。核心洞察：OSS 原生兼容 S3 API，无需引入新 SDK，只需在账户模型中引入 `CloudProvider` 枚举，让现有 `R2Service`（底层 AWS S3 SDK）同时服务于 R2 和 OSS。

### 改动文件

#### 新增文件
- `OwlUploader/CloudProvider.swift` — `CloudProvider` 枚举（`.r2` / `.oss`）和 `OSSRegion` 区域定义

#### 修改文件

| 文件 | 改动内容 |
|------|---------|
| `R2Account.swift` | 新增 `provider`、`ossRegion` 字段；更新 Codable 向后兼容；`isValid()` 按供应商分派；`keychainAccountIdentifier` 按供应商生成 |
| `R2Service.swift` | `createS3Client()` 按 provider 构造不同 endpoint/region；新增 `validateOSSEndpoint()`；`purgeCDNCache()` 跳过非 R2 供应商；`generateFileURL/BaseURL` 使用通用 endpoint |
| `R2AccountManager.swift` | `deleteAccount()` 按 provider 条件删除 Cloudflare API Token |
| `SidebarView.swift` | `AddAccountSheet` 新增 Provider 选择器、OSS Region Picker、按供应商动态表单；`AccountRow` 按 provider 显示不同图标 |
| `AccountSettingsView.swift` | `EditAccountSheet` 按 provider 显示/隐藏字段（Account ID、OSS Region、CDN 配置）；`AccountsTabView` 展示供应商信息 |
| `AppStrings.swift` | 新增 `provider`、`ossRegion`、`regionInfo` 本地化键 |
| `docs/features/01-account-configuration.md` | 更新为多云配置文档 |

### 架构决策

1. **复用 AWS S3 SDK**：OSS 兼容 S3 API，无需新 SDK
2. **`CloudProvider` 枚举**：扩展性好，未来添加新供应商只需新增 case
3. **向后兼容**：旧数据无 `provider` 字段时默认为 `.r2`
4. **CDN Purge 条件化**：通过 `provider.supportsCDNPurge` 控制
5. **OSS Region 内置列表**：硬编码常用区域，基于阿里云官方文档

## Bug 修复：OSS SecondLevelDomainForbidden 错误

### 问题

OSS 返回 `SecondLevelDomainForbidden` 错误：
> The bucket you are attempting to access must be addressed using OSS third level domain.

**根因**：AWS S3 SDK 使用自定义 endpoint 时默认使用 path-style 寻址（`endpoint/bucket-name/key`），但 OSS 要求 virtual-hosted-style（`bucket-name.endpoint/key`）。

### 修复

1. **`CloudProvider.swift`**：新增 `usePathStyle` 属性
   - R2 → `true`（path-style，endpoint 已含 accountID）
   - OSS → `false`（virtual-hosted-style，bucket 作为子域名）
2. **`R2Service.swift` `createS3Client()`**：配置 `s3Config.forcePathStyle = account.provider.usePathStyle`

### 影响

- R2 行为不变（继续使用 path-style）
- OSS 使用 virtual-hosted-style，符合阿里云要求

## Bug 修复：OSS Copy Link URL 格式错误

### 问题

Copy Link 生成的 URL 仍然是 path-style 格式：
`https://oss-cn-hangzhou.aliyuncs.com/bucket-name/key` ← 错误

应为 virtual-hosted-style：
`https://bucket-name.oss-cn-hangzhou.aliyuncs.com/key` ← 正确

### 修复

1. **`R2Service.swift`**：新增 `buildStorageURL()` 私有方法，根据 `usePathStyle` 生成不同格式 URL
   - path-style → `endpoint/bucket/key`（R2）
   - virtual-hosted → `bucket.host/key`（OSS）
2. `generateFileURL()` 和 `generateBaseURL()` 均改用 `buildStorageURL()` 统一生成

## Bug 修复：OSS 文件夹重命名失败 + CDN Purge 误报

### 问题 1：文件夹重命名报 UnknownAWSHTTPServiceError

**根因**：两层问题叠加
1. OSS 对文件夹标记对象（key 以 `/` 结尾的零字节对象）不支持 CopyObject
2. `renameObject` 只复制文件夹标记，不处理子文件（内容不会被重命名）

### 修复

1. **`R2Service.swift` `renameObject()`**：检测文件夹（key 以 `/` 结尾）时委托给新增的 `renameFolder()` 方法
2. **新增 `renameFolder()`**：列出旧前缀下所有子对象，逐个复制到新前缀，然后删除旧对象
3. **新增 `copySingleObject()`**：统一 copy 逻辑
   - 文件夹标记（key 以 `/` 结尾）→ PutObject 创建空对象（兼容 OSS）
   - 普通文件 → CopyObject + URL 编码（兼容特殊字符）
4. **`moveObject()` 和 `moveFolder()`**：复用 `copySingleObject()`，统一行为

### 问题 2：OSS 账户上传后误报 CDN purge 失败

**根因**：`UploadQueueManager.performUpload()` 无条件收集 CDN purge URL，OSS 账户的 `purgeCDNCache()` 返回 `false` 被误判为失败

### 修复

1. **`R2Service.swift`**：新增 `supportsCDNPurge` 只读属性
2. **`UploadQueueManager.swift`**：收集 CDN purge URL 前先检查 `r2Service.supportsCDNPurge`
